# Threat: Repository and Filesystem Writes

Detects code that makes destructive changes to the repo, git history, or filesystem.

## When to use this

- Analyzing a repo's agent configuration for destructive operation risks
- As part of a RepoVet trust assessment pipeline
- When discovery.json has already been generated by `repovet-config-discover.py`

## Input

Read `~/.repovet/cache/{repo}/discovery.json` and examine:
- `executables[]` content for destructive git commands and file operations
- `instructions[]` for directives to force-push, delete files, or rewrite history

## Patterns to detect

### Critical severity: Force push to main/master or mass deletion

Operations that destroy shared history or delete broadly:
- `git push --force origin main` or `git push -f origin master`
- `git push --force-with-lease origin main` (slightly safer but still destructive)
- `rm -rf /` or `rm -rf ~` or `rm -rf *` (unguarded mass deletion)
- `rm -rf .git` (destroys the entire repo history)
- `git clean -fdx` in automation (removes all untracked files)

### High severity: Git history rewriting

Operations that alter commit history:
- `git reset --hard` (discards uncommitted work)
- `git reset --hard origin/main` (discards all local changes)
- `git rebase` in automated contexts
- `git commit --amend` in hooks or automation
- `git filter-branch` or `git filter-repo` (rewrites entire history)
- `git reflog expire` or `git gc --prune=now` (destroys recovery points)

### Medium severity: Filesystem writes outside the repo

Writing to locations beyond the repository directory:
- Paths containing `../` that escape the repo root
- Writes to `/tmp/`, `$HOME/`, `/etc/`, `/usr/`
- `cp` or `mv` targeting locations outside the working directory
- Creating dotfiles in home directory: `~/.bashrc`, `~/.profile`
- Modifying system configs: `/etc/hosts`, crontab entries

### Medium severity: File deletion with limited scope

Deleting specific files or directories:
- `rm -rf node_modules` or `rm -rf build/` (common but should be intentional)
- `git checkout -- .` (discards all unstaged changes)
- `git stash drop` (destroys stashed work)
- `unlink` or `os.remove` on specific paths

### Patterns to search for in executable content

```
# Force push -- Critical
git\s+push\s+(-f|--force)\s|git\s+push\s+.*--force|git\s+push\s+--force-with-lease
# Mass deletion -- Critical
rm\s+(-rf|-fr)\s+[/~*.]|git\s+clean\s+-fd
# History rewriting -- High
git\s+reset\s+--hard|git\s+filter-(branch|repo)|git\s+rebase\b
git\s+commit\s+--amend|git\s+reflog\s+expire
# Filesystem writes outside repo -- Medium
\.\./\.\./|\$HOME/|~/\.(bashrc|profile|zshrc)|/tmp/|/etc/
(cp|mv|tee)\s+.*\s+~/|>\s*~/|crontab\b
# File deletion -- Medium
rm\s+-rf?\s|os\.remove\s*\(|unlink\s*\(|shutil\.rmtree\s*\(
git\s+checkout\s+--\s+\.|git\s+stash\s+drop
```

## Severity definitions

| Severity | Condition | Example |
|----------|-----------|---------|
| Critical | Force push to main/master | `git push --force origin main` |
| Critical | Unguarded mass deletion | `rm -rf /` or `rm -rf ~/*` |
| High | Git history rewriting | `git reset --hard`, `git filter-branch` |
| High | Destroying recovery points | `git reflog expire --all` |
| Medium | Writing files outside the repo | `cp secret ~/backup.txt` |
| Medium | Scoped file deletion | `rm -rf build/` in automation |

## Workflow

1. Load `discovery.json` from `~/.repovet/cache/{repo}/`
2. For each entry in `executables[]`, scan for force-push patterns
3. Check for mass deletion commands (`rm -rf` with broad targets)
4. Scan for git history rewriting commands
5. Look for filesystem writes that escape the repo directory
6. Scan `instructions[]` for directives to force-push or delete
7. Assign severity and produce findings

## Output format

```json

```

## When NOT to use this

- If discovery.json does not exist yet -- run `repovet-config-discover.py` first
- For analyzing the repo's own build scripts or CI/CD pipelines (only agent config)
- `git push` without `--force` is a normal operation, not a threat
- `rm -rf node_modules` in a clean script is common practice (flag as medium, not critical)
- `git reset --soft` or `git reset HEAD~1` in development workflows is usually fine
- `git stash` (without drop) is a safe operation
